## 들어가며

보안 최종 프로젝트의 두 번째 멘토링이 마무리되었다.

1주 차에 전체적인 기술 스택과 기획을 견고하게 다져두었지만, 2주차에 진행해야 할 과제는 적지 않았다.  
2차 멘토링 전까지 4일이라는 짧은 시간 내에 웹, 모바일 앱, 그리고 AWS 기반의 인프라까지 최소 70% 이상, 목표로는 100% 구현을 완료해야 하는 빠듯한 상황이었다.

웹, 모바일, 인프라 세 파트로 나누어 병렬적으로 기획을 진행했던 1주 차의 경험을 바탕으로, 구현 역시 각 팀이 병렬로 진행한다면 물리적인 목표치는 달성할 수 있을 것이라 판단했다.  
하지만 팀장으로써 가장 큰 고민은 단순히 기한 내에 코드를 완성하는 것이 아니었다.  
구현이 완료된 이후, 모든 팀원이 자신이 맡지 않은 타 도메인의 구현 상세를 완벽히 이해하고 취약점 진단 시나리오를 도출할 수 있어야 한다는 점이 더 중요했다.  
팀원들이 체력적으로 무리하지 않는 선에서 개발 목표를 달성하고, 나아가 서로의 지식을 동기화할 수 있도록 일정을 설계하는 데 집중했던 2주 차의 타임라인을 회고해 본다.

---

## 2주 차 타임라인 (Day 1 ~ Day 4)

1순위 목표인 빠른 구현과 2순위 목표인 팀원 간의 구현 사항 스터디를 모두 충족하기 위해, 3일 동안은 개발에만 집중하고 마지막 4일 차에 전체 브리핑 및 지식 동기화 시간을 가지는 것으로 전체 일정을 수립했다.

### Day 1: 개발 병목 현상 제거와 스켈레톤 코드 구축

웹과 모바일 팀의 경우, 아무런 코드베이스가 없는 초기 상태에서는 팀원들 간의 GitHub 브랜치 협업이 어려울 수밖에 없다.  
각 개발파트 PM이 뼈대를 잡아주며 팀원들은 본인이 맡은 모듈 단위의 개발에만 즉각적으로 몰입할 수 있는 환경을 조성하기 위해 각 팀의 파트장들이 AI를 적극 활용하여 기획된 설계에 맞춘 기본 코드베이스와 스켈레톤 코드를 빠르게 구성했다.

기획 단계에서 이미 요구사항이 명확히 정의되었던 인프라 팀은 1일 차에 아키텍처 구축을 모두 완료하는 속도를 보여주었다.

### Day 2: 병렬 모듈 개발과 AI를 활용한 생산성 극대화

2일 차부터는 구성된 코드베이스를 바탕으로 각 팀원들이 병렬 구현에 돌입했다.  
웹 팀(4명)은 API, DB, Web Frontend로 세부 역할을 나누어 빠르게 구현을 진행했다.

모바일 팀은 한 명이 AI 도구를 활용해 UI와 API 연동 코드를 고속으로 작성하는 동안, 나머지 3명은 루팅 및 Frida 탐지 등 보안 솔루션 적용과 우회 테스트, 그리고 WebView 기반의 JS Bridge 및 XSS 취약점 시나리오 테스트 등 보안 기능 구현에 집중하며 개발과 단위 테스트를 동시에 진행했다.

인프라 팀 역시 구축된 아키텍처 위에서 VPN 서버 연결과 더미 데이터 세팅 등 세부 설정을 고도화했다.

### Day 3: 통합 테스트 및 인프라 자동화(Terraform) 도입

3일 차에는 전체 기능을 완성하고 AWS 환경에 올려 동작을 검증하는 데 주력했다.

웹 서비스의 세부 기능 연동이 완료되었고, 모바일 역시 보안 솔루션 고도화와 API 변경 사항을 반영했다.

특히 인프라 팀은 기존에 Shell Script로 구성했던 인프라 구축 자동화 로직을 Terraform 기반으로 마이그레이션하며 시스템의 안정성과 재사용성을 한 단계 끌어올리는 성과를 달성했다.

### Day 4: 사일로(Silo) 현상 해결

팀원들의 역량과 노력 덕분에 단 3일 만에 웹, 모바일, 인프라 구축이 거의 완료되었다.  
하지만 팀장으로서 우려했던 문제, 즉 각 팀이 병렬로 속도를 내다보니 자신이 개발한 파트 외에는 시스템 전체를 파악하지 못하는 사일로 현상이 발생할 조짐이 보였다.

보안 취약점 진단과 모의해킹은 단일 모듈이 아닌, 시스템 전체의 아키텍처와 데이터 흐름을 이해할 때 비로소 의미 있는 시나리오가 도출된다.

이를 해결하기 위해 4일 차에는 개발을 멈추고 각 팀별로 세부 디테일을 타 팀에게 브리핑하는 일정을 마련했다.  
하지만 단발적인 브리핑만으로는 방대한 코드를 완벽히 숙지하기 어렵다고 판단했다.

이에 Claude Code를 활용해 웹, API 서버, 모바일 앱의 전체 코드베이스를 분석하고, 파이프라인과 상세 로직을 모두가 쉽게 열람할 수 있도록 Vercel 기반의 Study Documents 사이트로 구축하여 배포했다.

단순히 소스 코드를 던져주고 읽어보라고 요구하는 것이 아니라, 팀원들이 보안 진단이라는 본연의 업무에 역량을 100% 쏟을 수 있도록 최적의 분석 환경을 세팅해 주는 것이 중요하다고 생각했다.

## 2차 멘토링 피드백

2주차 멘토링에서는 단순한 취약점 나열을 넘어, 로직의 결함을 파고드는 실무적인 보안 디테일에 대해서도 깊이 있는 멘토링이 진행되었다.

### AWS 아키텍처 피드백

인프라 파트는 Terraform을 도입하여 VPC, ALB, EC2, RDS 등의 인프라 구축과 GitHub Actions, CodeDeploy를 연동한 CI/CD 파이프라인까지 견고하게 완성해 둔 상태였다.

하지만 2차 멘토링에서 아키텍처와 구성도를 리뷰하던 중, 멘토님으로부터 "현재 클라우드 IAM 권한이 SSM 등 서버 한 대에만 영향을 미치도록 너무 완벽하고 안전하게 구성되어 있어서, 역설적으로 해킹 시나리오의 폭이 너무 좁습니다." 라는 피드백을 받았다. 

멘토님은 단순히 AWS EC2 내에 접근하여 데이터를 탈취하는 것 이외에도, 취약한 권한 설정을 통해 AWS 계정 전체를 장악하여, 공격자가 원하는대로 클라우드 전체를 통제할 수 있도록 구성하는 방식을 제안해주셨다.

### JWT 토큰 기반 RBAC

JWT 토큰 기반의 인증/인가 구성에서도 멘토님의 조언이 있었다.  
현재 API는 JWT 기반 RBAC(역할 기반 접근 제어)로 구현되어 있는데, 로그아웃을 하더라도 JWT의 유효기간(30분)이 남아있다면 해당 토큰을 재사용할 수 있다는 취약점이 존재했다.

멘토님은 이를 해결하기 위한 방안을 생각해보면 좋을 것 같다는 조언을 주셨다.  
이후 팀원들과 서버 프레임워크 단에서 이를 강제로 만료시키는 것이 어렵기 때문에, 클라이언트 측에서의 토큰 삭제, 서버의 블랙리스트 관리, 혹은 만료 시간을 극단적으로 짧게 가져가는 방안 등 아키텍처 레벨의 복합적인 방어 대책을 논의하며 팀원들과 인증 체계의 한계를 깊이 생각해볼 수 있었다.

### 마이페이지 보안과 UX

개인정보가 집중된 마이페이지 진입 시, 현재 비밀번호를 재확인하는 로직이 누락되어 있었다.  
하지만 정보 수정 시 매번 비밀번호를 입력하게 만드는 것은  UX 저하를 야기하기에 보안성과 편의성 사이의 트레이드오프를 어떻게 조율할 것인가에 대해 생각해 보라는 멘토님의 조언이 있었다.

### Command Injection & Race Condition

인프라 공격의 방향성도 수정되었다. 기존에 기획했던 웹쉘 업로드로 인한 인프라 진입 시나리오는, S3 버킷에 업로드를 진행하는 설계로 인해 무산되었고, 팀원들과의 아이디어 브레인 스토밍을 통해 이메일 서버를 통한 Command Injection 시나리오를 삽입하는 방향으로 논의가 이루어졌다.

또한, 최근 버그바운티 트렌드에서 Race Condition 취약점이 자주 발생한다는 이야기를 해주시며, 앞으로 있을 시나리오 구성에 참고해 보라는 조언을 주셨다.

## 마치며

기본적인 구현이 완료된 현재 우리 팀의 다음 과제는 명확한 기준을 세우고 그에 맞춰 시스템을 진단하는 것이다.

멘토링을 통해 논의한 결과, 웹 부문은 주요정보통신기반시설 취약점 분석 가이드를 참고하고, 모바일 부문은 OWASP MASTG, 전자금융기반시설 보안 취약점 평가기준 등을 참고하여 진단 기준 수립하기로 결정했다.  
자체적인 진단 기준을 수립하더라도 그 뿌리가 되는 기반 문서를 명확히 해야만, 추후 작성될 진단 보고서의 객관성과 신뢰성을 평가자에게 증명할 수 있기 때문이다.

단 4일 만에 목표 구현율을 달성해 준 팀원 덕분에, 3주차에는 보안이라는 본질에 집중할 수 있는 환경을 마련할 수 있었다.

다음 주 3차 멘토링 전까지 우리의 최우선 목표는 공식 취약점 진단 기준을 참고하여 취약점 진단 기준을 수립하고, 수립된 기준에 따라 전체 서비스에 대해 세부 검증을 마친 뒤 취약점 진단 점검 결과 보고서를 도출해 내는 것이다.  
웹, 모바일 진단 기준 수립 및 진단을 최우선 목표로 진행하되 시간이 충분하다면, OS와 클라우드 취약점 진단까지 진행하는 것이 최종 목표이다.
